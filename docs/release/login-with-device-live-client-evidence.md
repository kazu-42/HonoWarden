## Scope

This evidence proves the narrow synthetic login-with-device path between the
official Desktop and browser-extension clients. It covers a clean Desktop
password login, owner notification and approval, one-time browser token
exchange, and empty-vault rendering. It does not authorize production use or
real vault data, and it does not claim a full item lifecycle, TOTP, attachment,
or live-regression pass.

No password, access code, access token, refresh token, fingerprint phrase,
email address, wrapped account key, private key, or encrypted response key is
recorded in this document or its screenshots.

## Protocol Repairs

The live run added regression coverage for four official-client requirements:

1. accept both `POST /api/auth-requests` and the official extension's trailing
   slash form, `POST /api/auth-requests/`;
2. return `masterKeyEncryptedUserKey` in sync user-decryption metadata;
3. send pending request type `15` through the owner-scoped authenticated hub;
4. validate and proxy request-scoped anonymous WebSockets, then send response
   type `16` to the official `AuthRequestResponseRecieved` SignalR target.

The anonymous WebSocket token is treated as bearer material. The Worker opens
the socket only for an existing, unexpired request with an owner, limits the
token length, and shards the Durable Object under an `auth-request:` namespace
so it cannot collide with owner-scoped notification objects. Notification
payloads contain only request ID, owner user ID, and an allowlisted type `15` or
`16`; encrypted key delivery remains on the authoritative HTTP response.

## Residual Risk

- Staging deployment replaced active Durable Object sockets, and the official
  clients used their reconnect behavior; a client restart/reload provided a
  deterministic immediate reconnect during this run.
- The Desktop logs an upstream `sshagent.clearkeys` missing-handler message at
  startup. The same process subsequently completed login, sync, notification
  approval, and vault rendering, so it is recorded as non-blocking client noise
  rather than a HonoWarden auth failure.
- Repeated notification resend attempts can leave older requests visible until
  their fixed expiry. Fingerprint matching remains mandatory, and superseding
  older pending requests should be tracked separately rather than hidden in
  this evidence.
