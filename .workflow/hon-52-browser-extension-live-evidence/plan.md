# HON-52 browser extension live evidence

## Goal

Capture redacted live evidence for the tracked browser extension release
(`browser-v2026.6.1`) against a local synthetic HonoWarden Worker, then update
the compatibility matrix and release evidence without recording any real vault
data or secrets.

## Success Criteria

- Verify the official browser extension asset digest before use.
- Run the extension in a clean local browser profile, configured for the local
  self-hosted HonoWarden endpoint.
- Bootstrap a synthetic account whose key material is generated by official
  client-compatible primitives, not hand-rolled test-only crypto.
- Record login, initial sync, at least one item mutation path, post-mutation
  sync, and refresh/session continuity when the client exposes the flow.
- Store only redacted evidence in the repository. Passwords, tokens, private
  keys, wrapped keys, raw request bodies, and raw response bodies stay under
  ignored `test/.tmp/`.
- Promote only the browser extension matrix row that the evidence actually
  proves. Broader desktop/mobile/TOTP claims remain separate issues.

## Current Context

- HON-39 is merged and closed; `main` is at `56a1285`.
- HON-24 remains open because production D1 has no stored inquiry message yet.
  The worker route exists, but no matching `inquiry-smoke@honowarden.com`
  event has reached Email Routing in the last 24 hours.
- The tracked browser extension release is `browser-v2026.6.1`.
- The downloaded Chrome extension zip under `test/.tmp/` has SHA-256
  `fcd29c5971d9b218ad9159717a19c38cca5150f2a0aa909ddf805bd7695d097e`.
- Existing CLI live evidence proves a local synthetic CLI flow only. Browser,
  desktop, mobile, TOTP, and full live regression remain unpromoted.

## Constraints

- No production Cloudflare writes for this issue.
- No real vault data, production identities, real passwords, tokens, private
  keys, forwarding destinations, or mailbox content in committed artifacts.
- The browser extension must be treated as a real upstream client binary. Avoid
  patching the extension bundle unless the resulting evidence is explicitly
  marked as patched-client evidence, which is not the target here.
- Unsupported feature failures must be recorded honestly instead of hidden by
  local fallback behavior.

## Risks

- The extension may reject a bootstrap payload unless user-key wrapping exactly
  matches upstream client crypto.
- Chrome managed storage/self-hosted environment configuration can be brittle
  for unpacked extensions because the extension id is assigned at load time.
- Local `wrangler dev` TLS and gzip behavior can differ from production; if the
  existing compression-stripping proxy is needed, the evidence must state that
  clearly.
- A narrow login/sync smoke is not a full compatibility claim. The matrix must
  remain conservative unless regression coverage is complete.

## Approval Required

No additional approval is required for local synthetic evidence. Escalate before
any production writes, real mailbox inspection, or use of non-synthetic vault
data.

## Work Packets

1. Asset provenance and source-map orientation.
2. Synthetic key/account generation using official client-compatible crypto.
3. Local Worker, local D1, and optional HTTPS proxy startup.
4. Browser extension automation in a clean profile.
5. Redacted evidence capture, matrix/doc update, tests, PR, and Linear closeout.

## Integration Policy

Commit reviewable documentation and deterministic helper code only. Keep
downloaded upstream assets, generated profiles, local D1 state, tokens, keys,
screenshots with secret-bearing UI, and raw browser logs under ignored
`test/.tmp/`.

## Verification

- Targeted evidence helper tests, if helper code is added.
- `pnpm compat:test`
- `pnpm release:gate -- --strict`
- `pnpm brand:scan`
- Full `pnpm test` if product docs or matrix shape changes touch shared
  compatibility gates.

## Reusable Artifacts

- Redacted browser extension evidence under
  `docs/release/live-client-evidence/` or another established release-evidence
  path.
- Optional local-only runner under `test/.tmp/` while discovering the flow.
- Reviewable reusable script only if the runner is deterministic, secret-safe,
  and valuable for future client evidence issues.
