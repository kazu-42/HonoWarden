# HON-30 attachment storage

## Goal

Implement encrypted attachment storage support for the HonoWarden vault API.
Attachment bytes remain opaque client-encrypted data; the server owns only
authorization, metadata, object-key generation, and lifecycle coordination.

## Success Criteria

- D1 stores attachment metadata scoped by `user_id` and `cipher_id`.
- R2 object keys are generated by the server and do not contain plaintext user,
  cipher, email, vault, or filename identifiers.
- Authenticated users can upload, download, and delete attachments only for
  their own ciphers.
- Cipher read/list/sync responses include attachment metadata.
- Permanent cipher deletion removes attachment metadata through DB lifecycle and
  the API deletion path removes the R2 object for the deleted attachment.
- Migration, repository, HTTP route, compat fixture, and backup/restore docs are
  updated and tested.

## Current Context

- Current app has placeholder unsupported routes for `/api/attachments*` and
  `/api/ciphers/:id/attachment*`.
- `VAULT_OBJECTS` R2 binding already exists in `src/bindings.ts` and
  `wrangler.jsonc`.
- Backup CLI already supports D1 export plus R2 object get/put/list using a
  reviewed object list or remote R2 listing.
- HON-57 is external audit work and HON-24 is waiting on a live email smoke; HON-30
  is the next actionable implementation issue.

## Constraints

- Do not run production migrations, deploys, or live object mutations in this
  workflow.
- Do not inspect, decrypt, log, or synthesize real vault contents.
- Keep R2 object keys opaque and generated server-side.
- Preserve existing cipher encrypted JSON behavior; server metadata must not
  require plaintext client vault data.

## Risks

- Route shape may differ across official clients. Keep compatibility fixtures
  conservative and document live-client follow-up separately.
- D1 metadata and R2 object writes are not a cross-resource transaction. Upload
  must clean up best-effort after metadata failure and fail loudly when R2 put
  fails.
- Delete can fail after metadata delete but before R2 deletion; return a 503 for
  unexpected infrastructure failures so operators can retry using metadata or
  backup evidence.
- Backup/restore remains operator-driven, so acceptance is local behavior plus
  documented object-prefix inclusion, not a production backup execution.

## Approval Required

- Not required for local code, tests, docs, PR, and Linear issue updates.
- Required before applying remote D1 migrations, deploying Workers, or mutating
  production/staging R2 objects.

## Work Packets

- `schema-repository`: migration and attachment repository CRUD/list behavior.
- `api-routes`: upload/download/delete routes and cipher response injection.
- `fixtures-docs`: compat fixture flow, live evidence plan, backup docs, current
  state/limitation updates.
- `verification`: targeted tests, full tests, check/lint/format, workflow
  verifier, PR CI/readback.

## Integration Policy

Prefer small repository functions over embedding SQL in route handlers. Preserve
existing Hono error response conventions and use owner-scoped predicates for all
lookups. Treat R2 objects as opaque bytes and D1 rows as the source of metadata
authorization.

## Verification

- Add failing tests before implementation where feasible.
- Run targeted repository/app/compat/migration tests first.
- Run `pnpm test`, `pnpm check`, `pnpm lint`, `pnpm format`, and `git diff --check`
  before PR.
- After merge, read back PR CI, main CI, and Linear state before closing HON-30.
